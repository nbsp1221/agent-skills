# AI 개발 워크플로우 프롬프트

이 문서는 Codex, Claude Code 등에 복붙해서 사용하는 개인용 단계별 개발 워크플로우 프롬프트 모음이다.

## 기본 워크플로우

1. 제품 스펙 문서 작성
2. 제품 스펙 문서 검토
3. 테스트 스펙 문서 작성
4. 제품 스펙 문서 및 테스트 스펙 문서 검토
5. 기술 구현 계획 문서 작성
6. 기술 구현 계획 문서 검토
7. 기술 구현 시작 및 검증
8. 최종 검증 및 회고

## 문서 역할

- 제품 스펙 문서: 무엇을 왜 만들지 정의한다. 구현 방법은 최소화한다.
- 테스트 스펙 문서: 제품 스펙의 요구사항과 계약을 어떤 테스트로 검증할지 정의한다.
- 기술 구현 계획 문서: 제품 스펙과 테스트 스펙을 바탕으로 실제 코드베이스에서 어떻게 구현할지 정의한다.

## 프롬프트 목록

### 1. 제품 스펙 문서 작성

```text
지금부터 새로운 기능/작업에 대한 제품 스펙 문서를 작성해줘.

먼저 내가 제공한 요구사항, 대화 내용, 코드베이스 맥락, 기존 문서가 있다면 그것을 바탕으로 이 작업의 의도와 문제 정의를 파악해. 필요한 경우 코드베이스를 조사해서 현재 구조와 제약을 확인해도 돼. 이 단계의 목표는 구현 방법을 정하는 것이 아니라, 무엇을 왜 만들지 명확히 정의하는 것이야. 따라서 how보다 what/why에 집중해.

제품 스펙 문서에는 다음을 포함해.
- 배경과 문제 정의
- 목표
- 비목표
- 사용자 의도
- 핵심 요구사항
- 기능 요구사항
- 비기능 요구사항
- 주요 시나리오
- 엣지 케이스와 실패 시나리오
- 외부에서 보장해야 하는 계약
- 성공 조건
- 아직 열려 있는 질문

중요한 원칙:
- 구현 방법을 과하게 결정하지 마.
- 모호한 요구사항은 임의로 확정하지 말고 "열려 있는 질문"으로 분리해.
- 단, 요구사항의 의도를 바꾸지 않는 범위에서, 일반적인 best practices로 명확히 판단 가능한 내용은 스스로 정리해.
- 기존 코드베이스나 문서에서 이미 확정된 내용이 있다면 그것을 우선해.
- 제품 스펙은 이후 테스트 스펙 문서와 기술 구현 계획 문서의 source of truth가 될 수 있어야 해.

완료 후에는 제품 스펙 문서 요약과, 인간에게 확인이 필요한 질문을 따로 보고해.
```

### 2. 제품 스펙 문서 검토

```text
지금까지 작성된 제품 스펙 문서를 검토해줘.

너는 이번 검토의 오케스트레이터 역할을 맡아. `Subagent Orchestration` 에이전트 스킬을 사용해서 최소 3개 이상의 제3자 서브에이전트 리뷰를 받아. 각 서브에이전트에게는 필요한 최소한의 맥락만 제공하고, 서로 다른 검토 관점을 명확하게 부여해. 맥락을 과하게 제공하면 기존 문서의 방향에 갇혀 다른 문제를 보지 못할 수 있으니 주의해.

서브에이전트 리뷰 관점 예시는 다음과 같아.
- 요구사항 및 제품 의도 리뷰어: 제품 스펙이 사용자의 의도와 문제 정의를 정확히 반영하는지 검토
- 엣지 케이스 및 실패 시나리오 리뷰어: 누락된 예외, 경계 조건, 비정상 흐름 검토
- 아키텍처 및 구현 가능성 리뷰어: 제품 스펙이 실제 코드베이스에서 구현 가능한 수준으로 명확한지 검토
- 테스트 가능성 리뷰어: 제품 스펙이 테스트 가능한 계약과 성공 조건으로 표현되어 있는지 검토

리뷰 결과 중 제품 의도나 기능 범위를 바꾸지 않는 명백한 개선 사항, 일반적인 best practices에 따라 판단 가능한 사항, 문서의 의도를 더 명확하게 만드는 수정은 나에게 확인받지 말고 스스로 문서에 반영해.

단, 다음 사항은 임의로 결정하지 말고 따로 모아서 나에게 질문해.
- 제품 의도나 요구사항 자체가 바뀔 수 있는 사항
- 여러 선택지가 있고 trade-off가 큰 사항
- 인간의 우선순위 판단이 필요한 사항
- 기존 스펙의 범위를 넘어서는 사항

완료 후에는 다음을 보고해.
- 사용한 서브에이전트와 각 리뷰 관점
- 발견된 주요 이슈
- 문서에 직접 반영한 변경 사항
- 아직 인간의 판단이 필요한 질문
```

### 3. 테스트 스펙 문서 작성

```text
지금까지 작성된 제품 스펙 문서를 바탕으로 테스트 스펙 문서를 작성해줘.

먼저 지금까지 작성된 제품 스펙 문서를 다시 읽고, 해당 기능을 왜 만들려는지, 어떤 사용자 의도와 계약을 만족해야 하는지 파악해. 그다음 가능하다면 web search를 통해 테스트 설계 best practices를 조사하고, 웹 접근이 불가능하다면 현재 알고 있는 일반적인 테스트 설계 원칙을 명시한 뒤 그 기준을 반영해서 테스트 스펙 문서를 작성해.

내 테스트 원칙은 다음과 같아.

1. 테스트 코드는 실제 코드보다 더 중요한 자산으로 관리한다. 좋은 테스트는 구현 세부사항이 아니라 외부에서 보장해야 하는 계약을 검증해야 한다. 테스트 이름, 구조, 입력, 기대 결과는 인간과 AI가 쉽게 이해할 수 있어야 하며 유지보수 가능해야 한다.

2. 정상 케이스만 테스트하지 않는다. 스펙의 의도를 파악한 뒤, 사용자가 들어올 수 있는 다양한 경로와 입력을 평가해야 한다. 예를 들어 sort 함수를 테스트한다면 단순히 [1, 3, 5, 2, 6]만 확인하면 안 된다. 역순, 이미 정렬된 값, 음수, 중복값, 단일 값, 빈 값, 잘못된 입력 등 의미 있는 경계 조건과 엣지 케이스를 함께 고려해야 한다.

3. 유닛 테스트에서 끝내지 않는다. 유닛 테스트는 필요하지만, 실제 버그는 여러 모듈과 흐름이 얽히는 지점에서 자주 발생한다. 따라서 통합 테스트, 회귀 테스트, 계약 테스트, E2E 테스트 등 필요한 테스트 레벨을 함께 설계해야 한다. 테스트를 설계하는 과정에서 실제 코드의 구조와 아키텍처가 더 테스트 가능하게 개선되어야 한다.

중요한 원칙:
- 테스트 스펙은 제품 스펙의 의도와 범위를 벗어나지 않아야 한다.
- 테스트를 이유로 새로운 제품 요구사항을 임의로 추가하지 마.
- 모든 테스트 레벨을 억지로 포함하지 말고, 제품 스펙의 리스크와 기능 특성에 따라 필요한 테스트 레벨을 선택하고 그 이유를 설명해.
- 테스트는 구현 세부사항이 아니라 외부에서 관찰 가능한 계약과 동작을 검증해야 한다.
- 모호한 테스트 기대값은 임의로 확정하지 말고 "열려 있는 질문"으로 분리해.

테스트 스펙 문서에는 다음을 포함해.
- 테스트 대상 기능의 의도와 핵심 계약
- 테스트 범위와 비범위
- 유닛 테스트 시나리오
- 통합 테스트 시나리오
- 필요하다면 E2E / 회귀 / 계약 테스트 시나리오
- 정상 흐름, 실패 흐름, 엣지 케이스, 경계 조건
- 테스트 데이터와 fixture 전략
- mock / stub / fake 사용 기준
- 테스트가 검증해야 하는 것과 검증하지 말아야 하는 것
- 테스트 레벨별 우선순위
- 성공 조건
- 아직 열려 있는 질문

완료 후에는 테스트 스펙 문서의 요약과, 특히 중요하다고 판단한 테스트 케이스를 보고해.
```

### 4. 제품 스펙 문서 및 테스트 스펙 문서 검토

```text
지금까지 작성된 제품 스펙 문서와 테스트 스펙 문서를 함께 검토해줘.

이 단계는 구체적인 기술 구현 계획 문서를 작성하기 전에 방향이 올바른지 확인하는 단계야. 방향을 처음부터 잘못 잡으면 이후 구현이 엉뚱한 방향으로 진행될 수 있으니, 제품 스펙과 테스트 스펙이 서로 일관되고 충분히 구체적인지 검토해.

너는 이번 검토의 오케스트레이터 역할을 맡아. `Subagent Orchestration` 에이전트 스킬을 사용해서 최소 3개 이상의 제3자 서브에이전트 리뷰를 받아. 각 서브에이전트에게는 필요한 최소한의 맥락만 제공하고, 서로 다른 검토 관점을 명확하게 부여해. 맥락을 과하게 제공하면 기존 문서의 방향에 갇혀 다른 문제를 보지 못할 수 있으니 주의해.

서브에이전트 리뷰 관점 예시는 다음과 같아.
- 제품 및 요구사항 정합성 리뷰어: 제품 스펙이 사용자 의도와 문제 정의를 정확히 반영하는지 검토
- 테스트 품질 리뷰어: 테스트 스펙이 구현 세부사항이 아니라 외부에서 보장해야 하는 계약과 동작을 검증하는지 검토
- 엣지 케이스 및 실패 시나리오 리뷰어: 제품 스펙과 테스트 스펙에서 누락된 경계 조건, 실패 흐름, 비정상 입력을 검토
- 구현 리스크 리뷰어: 이후 기술 구현 계획 단계에서 문제가 될 수 있는 모호성, 과도한 범위, 아키텍처 리스크를 검토

검토 기준:
- 제품 스펙의 what/why가 명확한가
- 테스트 스펙이 제품 스펙의 의도와 요구사항을 충분히 커버하는가
- 정상 흐름뿐 아니라 실패 흐름, 엣지 케이스, 경계 조건이 포함되어 있는가
- 테스트가 구현 세부사항이 아니라 외부에서 관찰 가능한 계약과 동작을 검증하도록 설계되어 있는가
- 제품 스펙과 테스트 스펙 사이에 모순이나 누락은 없는가
- 기술 구현 계획 문서를 작성하기에 충분히 명확한가
- 불필요하게 범위가 커졌거나 모호한 부분은 없는가

리뷰 결과 중 제품 의도나 기능 범위를 바꾸지 않는 명백한 개선 사항, 일반적인 best practices에 따라 판단 가능한 사항, 문서의 의도를 더 명확하게 만드는 수정은 나에게 확인받지 말고 스스로 문서에 반영해.

단, 다음 사항은 임의로 결정하지 말고 따로 모아서 나에게 질문해.
- 제품 의도나 기능 범위가 바뀔 수 있는 사항
- 테스트 우선순위에 대한 인간의 판단이 필요한 사항
- 구현 비용과 품질 사이의 trade-off가 큰 사항
- 기존 문서만으로는 의도를 확정하기 어려운 사항
- 제품 스펙과 테스트 스펙 사이의 충돌을 인간이 결정해야 하는 사항

완료 후에는 다음을 보고해.
- 사용한 서브에이전트와 각 리뷰 관점
- 발견된 주요 이슈
- 문서에 직접 반영한 변경 사항
- 기술 구현 계획 문서 작성 전에 인간에게 확인이 필요한 질문
```

### 5. 기술 구현 계획 문서 작성

```text
지금까지 작성된 제품 스펙 문서와 테스트 스펙 문서를 바탕으로 기술 구현 계획 문서를 작성해줘.

이 문서는 제품 스펙의 what/why가 아니라, 실제 코드베이스에서 어떻게 구현할지에 대한 how 문서야. 어떤 기술 스택, 아키텍처, 디자인 패턴, 기존 코드 경로, 재사용 가능한 함수와 로직을 사용할지 명확히 작성해야 해.

기술 구현 계획 문서를 작성하기 전에 반드시 코드베이스 전수조사를 먼저 실시해. 코드베이스를 조사하면서 다음을 파악해.
- 현재 프로젝트 구조
- 관련 도메인 / 모듈 / 파일
- 기존 아키텍처와 디자인 패턴
- 이미 존재하는 canonical path
- 재사용 가능한 함수, 타입, 컴포넌트, 테스트 유틸
- 기존 테스트 방식과 검증 명령
- 수정 시 영향을 받을 수 있는 영역
- 우회하거나 중복 구현하면 안 되는 기존 경로

기술 구현 계획 문서에는 다음을 포함해.
- 구현 목표
- 구현 범위와 비범위
- 코드베이스 조사 결과
- 사용할 기존 코드 경로와 재사용 대상
- 변경할 주요 파일과 책임
- 데이터 흐름 / 제어 흐름
- 아키텍처와 디자인 패턴 선택 이유
- 테스트 구현 계획
- 마이그레이션이나 호환성 고려 사항이 있다면 그 내용
- 리스크와 대응 방안
- 구현 순서
- 성공 조건
- 실행해야 할 검증 명령

중요한 원칙:
- 제품 스펙과 테스트 스펙의 의도를 벗어나지 마.
- 기존 canonical path를 우회하지 마.
- 같은 목적의 새로운 병렬 경로를 만들지 마.
- 단순한 문제를 과도하게 추상화하지 마.
- 기존 코드베이스의 컨벤션과 아키텍처를 우선해.
- 테스트 가능성을 고려해서 구현 방향을 설계해.
- 모호한 기술 선택이나 trade-off가 큰 선택지는 임의로 확정하지 말고 "열려 있는 질문"으로 분리해.
- 단, 제품 의도나 기능 범위를 바꾸지 않는 일반적인 best practices, 코드 재사용, 구조 정리, 명백한 기술적 개선은 스스로 판단해 반영해.

완료 후에는 기술 구현 계획 문서의 요약과, 핵심 설계 결정, 주요 변경 예정 파일, 예상 리스크, 검증 계획, 인간에게 확인이 필요한 질문을 보고해.
```

### 6. 기술 구현 계획 문서 검토

```text
지금까지 작성된 기술 구현 계획 문서를 실제 구현에 들어가기 전에 검토해줘.

너는 이번 검토의 오케스트레이터 역할을 맡아. `Subagent Orchestration` 에이전트 스킬을 사용해서 최소 3개 이상의 제3자 서브에이전트 리뷰를 받아. 각 서브에이전트에게는 필요한 최소한의 맥락만 제공하고, 서로 다른 검토 관점을 명확하게 부여해. 맥락을 과하게 제공하면 기존 문서의 방향에 갇혀 다른 문제를 보지 못할 수 있으니 주의해.

서브에이전트 리뷰 관점 예시는 다음과 같아.
- 아키텍처 리뷰어: 기존 코드베이스 구조와 canonical path를 올바르게 따르는지 검토
- 구현 리스크 리뷰어: 구현 순서, 의존성, 마이그레이션, 부작용 리스크를 검토
- 테스트 전략 리뷰어: 테스트 계획이 제품 스펙과 테스트 스펙을 충분히 커버하는지 검토
- 단순성 리뷰어: 과도한 추상화, 중복 구현, 불필요한 복잡성이 없는지 검토

검토 기준:
- 제품 스펙과 테스트 스펙의 의도를 충실히 반영하는가
- 코드베이스 전수조사 결과가 기술 구현 계획에 제대로 반영되었는가
- 기존 canonical path를 재사용하는가
- 우회 구현이나 병렬 경로를 만들 위험은 없는가
- 구현 순서가 작고 검증 가능한 단위로 나뉘어 있는가
- 테스트와 검증 계획이 충분한가
- 성공 조건이 명확한가
- 실제 구현에 들어가도 될 만큼 모호성이 줄어들었는가

리뷰 결과 중 제품 의도나 기능 범위를 바꾸지 않는 명백한 개선 사항, 일반적인 best practices에 따라 판단 가능한 사항, 기술 구현 계획을 더 안전하고 명확하게 만드는 수정은 나에게 확인받지 말고 스스로 문서에 반영해.

단, 다음 사항은 임의로 결정하지 말고 따로 모아서 나에게 질문해.
- 구현 방식 선택에 인간의 우선순위 판단이 필요한 경우
- 제품 스펙이나 테스트 스펙의 범위를 바꿔야 할 가능성이 있는 경우
- 구현 비용, 품질, 일정 사이의 trade-off가 큰 경우
- 기존 코드베이스의 의도를 확정하기 어려운 경우
- 여러 기술적 선택지가 있고 장단점이 큰 경우

완료 후에는 다음을 보고해.
- 사용한 서브에이전트와 각 리뷰 관점
- 발견된 주요 이슈
- 문서에 직접 반영한 변경 사항
- 기술 구현 시작 전에 인간에게 확인이 필요한 질문
```

### 7. 기술 구현 시작 및 검증

```text
자, 이제 지금까지 작성한 문서를 바탕으로 실제 work 단계로 진입하자.

너는 이번 작업의 오케스트레이터 역할을 맡아. 먼저 지금까지 작성된 제품 스펙 문서, 테스트 스펙 문서, 기술 구현 계획 문서를 읽고 목표, 구현 범위, 성공 조건, 검증 방법을 파악해. 그다음 `Subagent Orchestration` 에이전트 스킬을 사용해 필요한 서브에이전트들을 역할별로 투입해.

반드시 아래 Anthropic 문서를 읽고, 그 문서의 long-running agent harness 원칙에 따라 서브에이전트를 사용해.

https://www.anthropic.com/engineering/harness-design-long-running-apps

이번 작업은 이미 문서로 범위와 성공 조건이 명시되어 있으므로, 문서 범위 안의 구현, 수정, 테스트 추가, 검증, 리뷰 반영은 인간 승인을 기다리지 말고 자율적으로 진행해. 인간이 병목이 되지 않도록 해.

사용할 수 있다면 goal 관련 도구(create_goal, get_goal, update_goal 등)를 사용해서 전체 목표와 작업 상태를 추적해.

최종 목표는 다음과 같아.
- 제품 스펙 문서에서 요구한 기능과 계약을 구현한다.
- 테스트 스펙 문서에서 요구한 테스트를 구현한다. 단, 특정 테스트를 그대로 구현할 수 없는 경우에는 동등한 검증 범위를 확보하고, 그 이유와 대체 검증 방법을 최종 보고에 명시한다.
- 기술 구현 계획 문서의 구현 범위와 성공 조건을 모두 만족한다.
- lint, typecheck, test, build 등 프로젝트에 필요한 기계적 검증을 모두 통과한다.
- 구현 완료 후 서브에이전트를 사용해 제3자 리뷰를 반드시 받는다.
- 리뷰에서는 문서대로 구현되었는지, 성공 조건을 만족하는지, 우회 구현이나 임시방편은 없는지 확인한다.
- 리뷰에서 나온 blocker 또는 important 이슈는 수정하고 다시 검증한다.

중요한 원칙:
- 기존 canonical path를 우회하지 마.
- 같은 목적의 새로운 병렬 경로를 만들지 마.
- 임시방편으로 테스트만 통과시키지 마.
- 제품 스펙, 테스트 스펙, 기술 구현 계획 문서의 범위를 벗어나지 마.
- 실패가 발생하면 증상, 원인, 수정, 재검증을 분리해서 진행해.
- 모든 구현, 검증, 리뷰 반영이 끝난 뒤에만 완료로 판단해.

최종 보고에는 다음을 포함해.
- 구현 요약
- 변경된 주요 파일
- 성공 조건별 충족 여부
- 실행한 검증 명령과 결과
- 서브에이전트 리뷰 요약
- 리뷰 이슈 반영 내역
- 남은 리스크나 후속 작업
```

### 8. 최종 검증 및 회고

```text
구현이 완료되었다면 최종 검증 및 회고를 진행해줘.

먼저 제품 스펙 문서, 테스트 스펙 문서, 기술 구현 계획 문서, 실제 구현 결과를 다시 읽고, 이번 작업이 처음 의도한 목표와 성공 조건을 만족했는지 확인해.

그다음 가능한 모든 기계적 검증을 다시 실행해.
- lint
- typecheck
- test
- build
- 프로젝트에 정의된 추가 검증 명령

`Subagent Orchestration` 에이전트 스킬을 사용해서 최종 제3자 리뷰를 받아. 이 리뷰는 구현자가 놓친 문제를 찾기 위한 리뷰이므로, 리뷰어에게는 필요한 최소한의 맥락만 제공하고 독립적인 관점으로 검토하게 해.

최종 리뷰 관점:
- 제품 스펙의 요구사항과 성공 조건을 모두 만족하는가
- 테스트 스펙의 핵심 시나리오와 계약이 충분히 검증되었는가
- 기술 구현 계획 문서와 실제 구현 사이에 불일치가 없는가
- 기존 canonical path를 우회하지 않았는가
- 임시방편이나 불필요한 복잡성이 남아 있지 않은가
- 유지보수하기 어려운 코드나 테스트가 없는가
- 문서와 구현 사이에 불일치가 없는가

blocker 또는 important 이슈가 발견되면 완료로 선언하지 말고 수정 후 다시 검증해.

회고 문서에는 다음을 포함해.
- 최종 구현 요약
- 성공 조건별 충족 여부
- 실행한 검증 명령과 결과
- 최종 리뷰 결과
- 수정한 리뷰 이슈
- 잘 된 점
- 아쉬웠던 점
- 다음 작업에서 개선할 점
- 남은 리스크
- 후속 작업 제안

완료 후에는 최종 회고 요약과, 아직 인간이 확인해야 할 사항이 있다면 따로 보고해.
```
